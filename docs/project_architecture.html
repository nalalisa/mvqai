문제 정의

Why This Project Exists
  • 기존 rule-based geometry 파이프라인은 해부학적 변이에 취약합니다.
  • AoV control point를 hard tube mask로 바꾸면 링이 두꺼워지거나 중간이 끊기기 쉽습니다.
  • 일반 voxel overlap loss만으로는 중심선 연결성과 닫힌 loop 구조를 충분히 보장하기 어렵습니다.
  • TEE 3D 데이터는 small object, anisotropy, clutter noise가 동시에 존재해 모델이 배경으로 붕괴하기 쉽습니다.

설계 원칙

Design Principles
Topology 값만 맞추지 않고 구조까지 맞춘다.
Reproducibility 실험 변경은 코드가 아니라 YAML에서 한다.
Extensibility AoV heatmap에서 MV/AoV multi-task 구조로 확장 가능해야 한다.

End-to-End 데이터 흐름

From Markup To Training Metrics
1. Raw Input

3D TEE volume와 mrk.json control point를 수집합니다.

2. Spline

닫힌 B-spline으로 control point를 부드럽게 보간합니다.

3. Heatmap

보간 점을 따라 3D Gaussian soft ring target을 렌더링합니다.

4. Manifest

train.json, val.json 에 image / target 경로를 기록합니다.

5. DataModule

MONAI transform으로 crop, flip, scaling, noise를 적용해 batch를 구성합니다.

6. Network

UNet, AttentionUnet, DynUNet 중 하나가 dense response volume을 예측합니다.

7. Objective

Weighted MSE와 Soft-clDice를 이용해 값과 topology를 동시에 맞춥니다.

모델 계층

Backbone Choices
Model Config When It Helps
UNet configs/model/unet.yaml 안정적인 baseline. heatmap regression을 시작할 때 기준점 역할을 합니다.
AttentionUnet configs/model/attention_unet.yaml 작고 중요한 구조를 강조하고, background clutter를 억제할 가능성이 큽니다.
DynUNet configs/model/dynunet.yaml 비등방성 축 해상도에 대응하기 좋고, 층별 비대칭 stride 설정이 가능합니다.

현재 래퍼 파일은 src/networks/monai_wrappers.py 와 src/networks/monai_nets.py 두 경로가 공존합니다. 둘 다 MONAI 모델을 얇게 감싸는 thin wrapper이며, 향후 하나로 통합하기 쉽습니다.

왜 Wrapper를 두는가

Abstraction Benefits
  • Hydra _target_ 경로를 프로젝트 내부 명명 규칙으로 통일할 수 있습니다.
  • 향후 multi-head output, shared encoder, task-specific decoder를 추가하기 쉬워집니다.
  • activation이나 post-processing 정책을 네트워크 레벨에서 통일할 수 있습니다.
  • MONAI 클래스를 직접 노출하는 대신 실험 아키텍처의 표준 진입점을 만들 수 있습니다.

Loss 설계

Value + Topology
Weighted / Masked MSEforeground voxel에 큰 가중치를 부여해 작은 링 구조가 배경에 묻히지 않게 합니다.
Soft-clDiceskeleton 기반 유사도를 이용해 끊어진 링과 구조 붕괴를 더 직접적으로 벌줍니다.
Combined LossL = alpha * MSE + (1 - alpha) * clDice 로 intensity fidelity와 connectivity를 같이 최적화합니다.

Hydra가 중요한 이유

Why Config Beats Hardcoded Experiments
  • 논문 실험 표를 만들 때 코드 분기를 만들지 않아도 됩니다.
  • loss 조합, sigma, backbone, precision 변경을 모두 커맨드라인 오버라이드로 처리할 수 있습니다.
  • 실험 출력 디렉토리가 분리되어 config snapshot과 결과를 함께 보존할 수 있습니다.
  • 같은 코드베이스로 baseline과 ablation을 빠르게 돌릴 수 있습니다.
python src/train.py -m loss.alpha=0.3,0.5,0.7 python src/train.py -m model=unet,attention_unet python src/train.py -m dataset.sigma=1.0,2.0,3.0

현재 코드 상태

Research Scaffold, Not Finalized Product

현재 저장소는 완전히 단일 경로로 정리된 production 코드라기보다, 연구를 빠르게 진행하기 위한 스캐폴드와 대안 구현이 함께 존재하는 상태입니다.

  • DataModule 두 경로가 공존합니다.
  • Loss 구현도 통합형과 분리형이 함께 있습니다.
  • Heatmap 생성기도 NRRD/NIfTI 두 포맷 흐름이 남아 있습니다.
  • Network wrapper 파일도 두 버전이 존재합니다.

정리 방향

Recommended Consolidation Path
  1. manifest 표준을 하나로 확정합니다.
  2. loss activation 정책과 mask 지원 여부를 기준으로 한 계열로 통합합니다.
  3. network wrapper를 하나의 파일로 정리합니다.
  4. heatmap 출력 포맷을 .nrrd 또는 .nii.gz 중 하나로 정합니다.
  5. 그 뒤 README와 HTML 문서를 표준 경로 중심으로 다시 단순화합니다.
